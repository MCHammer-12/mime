# Decisions

## 2026-06-08 — Reviewer dashboard is a separate process, open access, same Postgres
External reviewers (non-Redo, non-Replit) can't be added to the Redo workspace, and the Redo org forbids Public visibility on its Replit deploys. **Decision:** stand up a second deployment on Michael's personal Replit plan, public Autoscale visibility, same code, same Postgres. New env var `MIME_SURFACE=public_review` flips behavior at boot — `dispatchReviewerSurface` runs before any other route handler (including `checkBasicAuth`), serves only `/r/<token>/`, `/dashboard`, and `/api/r/*`, and 404s anything that looks like an admin/assist endpoint. Admin process unchanged when `MIME_SURFACE` is absent. Reviewer auth is **open access** — `requireReviewer` returns a synthetic `PUBLIC_REVIEWER` ({id:"public", name:"Reviewer"}) row when no cookie is present, and a migration seeds the row so the `stores.created_by_reviewer FK` is always satisfiable. Cookie-based per-reviewer scoping (via `/r/<token>/` handshake → `getReviewerByToken`) is still honored when present but isn't required. Alternatives considered: (a) Vercel deploy of just the reviewer surface — abandoned previously in PR #52 because admin invited Dennis/Toby to Replit instead; (b) drop Replit Private gate on the existing deploy — leaks the admin surface; (c) per-reviewer URL token gate — overkill for the trust level. Going forward, any new admin-only endpoint must be added to `MIME_SURFACE === "admin"` branch only; reviewer-side endpoints must live under `/api/r/*` and use `requireReviewer` + `getStoreForReviewer` ownership checks. 404 (never 403) on ownership mismatch so we don't leak the existence of other reviewers' stores.

## 2026-06-08 — Cross-process notes: refresh from DB on read, accept stale events
Admin (Repl A, Redo workspace, `MIME_SURFACE=admin`) and reviewer (Repl B, Redo Public, `MIME_SURFACE=public_review`) are separate Node processes sharing one Postgres. `setNote` writes to both in-memory `job.notes` and the `jobs.notes` JSONB column; reads on the OTHER process return whatever was in memory at boot. **Decision:** add `refreshJobNotesFromDb(jobId)` — a single `SELECT notes FROM jobs WHERE id = $1` that overwrites the in-memory cache for that job. Best-effort, swallows DB errors. Wired into `handleJobGet` (admin job detail), `handleJobBundle` (troubleshoot zip), and `handleReviewerJobsList` (sidebar polling). Not wired into event streams — events are append-only and the dashboards mostly care about status / summary which the DB already has at boot. Cross-process event sync would require Postgres LISTEN/NOTIFY or a polling event-tail; deferred until the missing-events scenario causes real pain. The reviewer-write-then-admin-read note path is the high-value case and is now covered.

## 2026-06-08 — 403 from Redo is permissions, not JWT refresh
`import-rpc.ts:postAtPath` had been throwing `RedoAuthExpiredError` on both 401 AND 403. `withFreshJwt` catches that error class and prompts the user for a fresh JWT to resume. For an EXPIRED token (401), that's correct. For a permission denial (403 — "Lacking required permissions"), pasting a fresh JWT for the same user just hits the same 403 again. Diagnosed live during the reviewer-import session: David (an external Redo user the reviewer was importing as) had a valid session but no admin role, so every `/marketing-rpc/createEmailTemplate` came back 403, and the reviewer got the JWT refresh prompt for every flow. Chrome MCP probe to the endpoint confirmed `{"error":"Lacking required permissions"}` with status 403. **Decision:** split: 401 still throws `RedoAuthExpiredError` (refreshable); 403 throws a regular `Error` with the response body included so the user sees "Lacking required permissions" instead of a useless JWT-refresh modal. Re-probe after Michael added David as admin returned 400 (validation failure on the empty probe payload), confirming permission propagation. Lesson: future Redo error-class additions need similarly explicit error-class splitting — don't bundle distinct HTTP status codes into one mime error class.

## 2026-06-08 — Parse `background` shorthand for color token (don't pass through whole string)
SHOC's "Email with Large Graphic" template used `background: #222222 url(https://...png) center center / auto repeat` on a wrapper element. `findAncestorBackgroundColor` returned the WHOLE shorthand string. That string landed on `sectionColor` in the exported template. Zod's `z.string()` accepted it; Redo's downstream pipeline (Mongoose save or one of the renderers) threw an uncaught exception trying to use it as a CSS color → 500 with body "Internal server error." Every saved-template import from SHOC failed identically. **Decision:** `extractCssColor(raw)` helper that strips `url(...)` tokens first, then matches in order of specificity: hex (`#fff` / `#ffffff` / `#ffffffaa`), functional values (`rgb()` / `rgba()` / `hsl()` / `hsla()`), then named colors filtered against a `NON_COLOR` set (so "repeat", "center", "auto", etc. don't get picked). Returns null when no color is present (pure `url(...)` background). 9/10 inline tests pass; the one near-miss is a `linear-gradient(red, blue)` shorthand which is vanishingly rare in Klaviyo's output. Alternative considered: only read `background-color` and ignore `background` shorthand entirely — rejected because that misses the SHOC-style "color via shorthand" case which is the common Klaviyo pattern.

## 2026-06-08 — CODE-template parser drops `imageSourceType` field entirely
Redo's `ImageType` enum was tightened from `{ url, upload, product }` to `{ upload, product }` only. mime's CODE-template parser was emitting `imageSourceType: ImageType.URL` on every image block. Result: 400 validation error on `createSavedEmailTemplate` with `{"imageSourceType":{"_errors":["Received url"]}}`. The field is optional in Redo's schema; absence means "URL-sourced image" which is the default semantics. **Decision:** drop the `imageSourceType` line entirely from the CODE-template image block emission. mime's own `ImageType.URL` constant stays in `renderer/types.ts` for the local renderer (only consumed locally). Going forward: any "lifted" image (uploaded to Redo, product reference) needs to explicitly emit `imageSourceType: "upload"` or `"product"`; the parsed-as-URL path leaves it absent. Same edge applies to any other Redo enum that gets tightened — never send a value the schema doesn't accept; optional fields default to null/absent.

## 2026-05-28 — `createSavedEmailTemplate` needs `_id` AND `team` on the embedded template
The saved-templates RPC (`createSavedEmailTemplate`, used for standalone + campaign imports since #38) embeds a `template` validated by the FULL `emailTemplateSchema`, and its Mongoose schema (`emailTemplateSchemaDefinition` in `redo/marketing/db/schema/src/EmailTemplate.ts`) marks both `_id` (ObjectId) and `team` (ObjectId, ref Team) as required. Unlike `createEmailTemplate` — whose input is `emailTemplateSchema.omit({_id})` and whose handler injects `team` from the JWT context — the saved-template handler injects `team` only on the OUTER wrapper doc, never on the embedded template. mime's `preparePayload` stripped both fields (`_id` "server generates", `team` "server sets from JWT"), which is correct for the EmailTemplate path but fatal for the saved path: stripping `_id` → 400 `template._id Invalid input` (#62 fix: keep `_id`); missing `team` → uncaught Mongoose ValidationError → 500 (#64 fix: `decodeJwtAud()` reads the JWT `aud` claim and injects `team` onto the embedded template for the saved path only). **Decision:** for `asSavedTemplate`, send a fully-populated embedded template (`_id` from mime's generated ObjectId, `team` from the JWT). The non-saved path still omits `_id`; Zod's `.omit({_id})` silently drops any extra `_id` we pass. Considered fixing the redoapp handler to inject `team` itself (cleaner, removes the asymmetry) — deferred as a redoapp change; mime works around it client-side for now.

## 2026-05-28 — Template `schemaType` is metadata; runtime resolution is the flow's job
When realigning a flow's email/SMS templates to match a changed trigger (the cascade in the `change-marketing-automation-trigger` skill), changing a template's `schemaType`/`category` does NOT change how it renders at send time. Dynamic fields resolve against the FLOW's schema instance (set by the trigger), not the template's own `schemaType` label. The label drives the builder's available-fields UI and the template-library classification, and removes a formal template-vs-trigger mismatch — but it won't populate data that isn't there. **Decision:** treat the realignment as a consistency/UX fix, always state this caveat to the user, and make it a mandatory part of the trigger swap (not an optional follow-up) so a flow is never left half-migrated. `updateEmailTemplate` is a `$set` partial (send only `schemaType`+`category`, never `sections`); `updateSmsTemplate` requires the full populated object (read via `getSmsTemplateByIdPopulated`, override two fields).

## 2026-05-26 — `<p>` style merge in text parser, not duplicate attribute
`parseTextBlock` baked `text-align` + `line-height` into the HTML by prepending a `style="…"` attribute to every `<p>` — but Klaviyo's inner paragraphs often already carry inline styles (e.g. `<p style="font-family: 'Futura', sans-serif">`). The output was two `style=` attributes on one tag, which is invalid HTML5: browsers and Quill keep ONLY the first attribute, silently dropping the inline `font-family`. Blackline Car Care template manifested this as Futura text rendering as a fallback patchwork after Austin manually changed the block-level font. **Decision:** merge our text-align/line-height into any existing `style="…"` attribute via a regex on `<p\b([^>]*)>`. Append-style means our values win on conflicts (which is correct — we sourced them from the outer wrapper that visually controls the paragraph). New `text.smoke.ts` covers bare `<p>`, mixed, trailing-semicolon, other-attributes-preserved cases. Alternative considered: stop baking into HTML entirely since `TextBlock.textAlign` / `lineHeight` exist as fields now; rejected because the inline injection also covers `<p>` tags inside non-wrapping content where the block-level field doesn't reach.

## 2026-05-26 — Unresolved fonts pause the import (preflight gate) instead of warning past
The 2026-04-16 design intent was "auto-upload + preflight-block fallback" for fonts; only the auto-upload half shipped. Unresolved fonts (Futura, custom Adobe/Linotype faces) used to emit a non-blocking `warn` in `server.ts`. Austin missed Blackline's warning, the template landed with `fontFamily: "Futura"` pointing at a font the brand kit didn't have, and the editor rendered fallbacks. **Decision:** convert the warn into a `needs_input` modal. Pauses the job, lists the missing families with `used by …` provenance, links the merchant to the brand-kit page. Two buttons via new `trueLabel`/`falseLabel`: "Continue (added them)" / "Import anyway". Re-checks the brand kit (`filterFontsNotInBrandKit`) before prompting so already-added fonts don't trigger spurious modals. `questionKey` scoped by the (sorted) font set so a later phase encountering a NEW family re-prompts rather than reusing a stale cached answer. Trade-off accepted: extra friction for merchants who DO want fallback rendering — they click "Import anyway" once and the cache reuses for the rest of the job.

## 2026-05-26 — Flow phase gains its own font upload (was previously absent)
`importFlowRpc` ran without any `uploadFontsForTemplates` call beforehand — flow-attached templates landed referencing fonts that may or may not have been added by a parallel template-phase upload. Discovered while wiring the preflight gate. **Decision:** add per-flow `uploadFontsForTemplates(placeholderTemplates.map(p => p.fullTemplate))` immediately before the `importFlowRpc` call, followed by the same `preflightUnresolvedFonts` gate. Per-flow (not batched across all flows) chosen for surgical scope; the upload helper already dedupes by family name + skips existing brand-kit entries, so repeated flows sharing fonts are no-ops on the wire. Pre-parsing all flows up-front to batch the upload would be cleaner but requires restructuring the flow loop — deferred.

## 2026-05-26 — Feedback notes gain `resolvedAt`, text edits auto-reopen
"Has feedback (N)" badge would keep climbing forever without a way to mark items addressed. Deleting the note loses the history (and the assistant's attribution). **Decision:** `StoredNote` gains optional `resolvedAt` (ISO timestamp) + `resolvedBy` (admin name). New `setNoteResolved(jobId, itemId, resolved, resolver?)` flips just those fields — preserves text + author + savedAt across resolve/reopen cycles. New `POST /api/jobs/:id/notes-resolve` endpoint. Plain-string legacy notes get migrated to the structured shape on first resolve. Text edits via the existing `setNote` path implicitly REOPEN any prior resolution — new content is fresh work that the resolution didn't account for. UI: per-item `✓ resolve` / `↺ reopen` button; row mutes (opacity-60, strikethrough) when resolved; `getJobNoteCount` and "select noted" drop resolved items so the to-do counter goes back to zero and re-exports skip already-shipped work. Job-route regex widened from `[a-z]+` to `[a-z-]+` to accept the hyphenated subpath.

## 2026-05-14 — Flow-imported templates inherit the flow's trigger schemaType
Templates created via `importFlowRpc` were hardcoded to `schemaType: "marketing_email"`. That's the most permissive type but the wrong one for any flow that needs trigger-specific dynamic variables — most visibly back-in-stock (`productName`, `productUrl`), but also cart_abandonment (`cartSubtotal`), order-tracking, Yotpo loyalty, reviews. The Redo email builder gates the variable picker on the template's schemaType. **Decision:** propagate `bundle.automation.schemaType` (set by `resolveTrigger` from the Klaviyo metric) onto both the full-template spread AND the blank fallback in `import-rpc.ts`. Standalone template imports (POST /api/templates) keep their `marketing_email` default — those are reusable, not flow-bound. Trade-off: pre-#61 imports still have wrong schemaType; would need re-import or per-template UI fix. Acceptable since impact is "back-in-stock vars don't appear in the editor" not "template is broken".

## 2026-05-12 — Heartbeat on /api/jobs/:id/stream survives Replit's idle-stream killer
Importer emits `needs_input` event → server blocks in `ctrl.prompt()` waiting for the modal pick. No bytes flow on the NDJSON stream during that wait. Replit's autoscale proxy kills idle streams in seconds; Chrome surfaces as `TypeError: network error`; modal answer never gets POSTed. Hit by Fairechild's "Review Request Klaviyo" flow ("Ready to review" Klaviyo metric isn't in METRIC_NAME_MAP → picker fired → stream died). **Decision:** 10s `{kind:"heartbeat",t:Date.now()}` heartbeat on `handleJobStream`, mirroring the existing pattern in `handleFlowsStream`. Client's `readNdjsonLines` treats unknown event kinds as no-ops so it's invisible to the UI. Alternative (server-sent events with native browser keepalive) considered — not worth the rewrite for one missing line.

## 2026-05-08 — Klaviyo "Started Checkout" → MARKETING_CART_ABANDONMENT (not CHECKOUT)
Strict semantics say "Started Checkout" → `marketing_checkout_abandonment` because that's the Klaviyo event for "entered checkout, didn't complete". But in practice merchants name their Klaviyo flow "Abandoned Cart" and use Started Checkout as its trigger; that's what Klaviyo's stock template does. Defiance Beauty hit this — flow was named "Defiance Beauty Abandoned Cart", we picked `marketing_checkout_abandonment`, user flagged it. **Decision:** map both `"started checkout"` and `"checkout started"` to `MARKETING_CART_ABANDONMENT`. Confirmed with Redo eng. Side effects: the `isCartAbandoned` auto-skip-condition fires instead of `isCheckoutAbandoned`. If we later need a separate signal for true "started Shopify checkout but didn't finish," it'd need a different Klaviyo metric (which most accounts don't have).

## 2026-05-08 — Migrated buttons use static `<storeUrl>/cart`, not `dynamic-variable checkoutUrl`
Originally mapped Klaviyo's four checkout-URL variables (`event.URL`, `event.CheckoutURL`, `event.extra.checkout_url`, `event.extra.responsive_checkout_url`) to `linkType:"dynamic-variable", schemaFieldName:"checkoutUrl"`. Defiance Beauty merchant reported buttons rendering as silently-hidden / broken. Asked Redo eng → confirmed `schemaInstance.checkoutUrl` is a Shopify Storefront cart URL (`/cart/c/{token}`), NOT Klaviyo-style session recovery URL (`/checkouts/c/...`). When the Storefront cart fetch fails (no Storefront access token, non-Shopify provider, fetch error), the field is `undefined`. Redo's button renderer treats undefined dynamic-var as `hideBlock = true` → returns null → button silently disappears from the email. **Decision:** rewrite all four patterns to `{ linkType:"web-page", buttonLink: "<storeUrl>/cart" }` when storeUrl is known (from KlaviyoAccount.websiteUrl). Trade-off: loses Klaviyo's customer-specific session token (no auto-login back to actual abandoned checkout); accept the downgrade because a working generic /cart link beats a silently-hidden button. The `MappedLink.dynamic-variable` variant removed from the type entirely (no callers emit it anymore). See `project_redo_checkout_url_resolution` memory for eng-team source notes. Open: revisit if Redo fixes the silent-hide path so dynamic-vars become reliable.

## 2026-05-08 — Merchant credentials live in Postgres (`stores` table), not browser localStorage
Browser-localStorage persistence meant: (a) diagnostics tooling couldn't read keys without operator paste; (b) keys vanished on cleared localStorage / new device; (c) JWT had to be re-pasted into each browser when it expired. **Decision:** migration 003 adds `stores(id, name, merchant_slug UNIQUE, klaviyo_key, redo_jwt, store_id, redo_server_base, ...)`. 5 endpoints (`GET/POST /api/stores`, `GET/PATCH/DELETE /api/stores/:id`) + a debug `POST /api/debug/resolve-template` that runs the same resolver path the flow importer uses, returning typed failures. `GET /api/stores/:id` returns the full unmasked record (the edit form needs it; threat model is "anyone with the admin route already has carte blanche via /api/run"). UI dashboard cards gain an edit pencil → setup modal in edit mode with JWT-expiry hint. localStorage fallback retained for local dev (no `DATABASE_URL`). Anthropic API key stays in env (not per-merchant). Subsequent admin-claim work in #41 layered an additional cookie gate.

## 2026-05-08 — Klaviyo "Ordered Product" collapses into Redo `order-placed`
Klaviyo's "Ordered Product" metric fires once per line item (so "Ordered Product zero times in last 7 days" means "didn't order any products"). Redo has only `ORDER_PLACED_SHOPIFY` (`"order-placed"`) — no per-line-item activity. **Decision:** map `"ordered product"` → `order-placed` in `METRIC_TO_ACTIVITY`. For zero-times conditions the semantics are equivalent (zero line items ⇔ zero orders); for non-zero counts the mapping is approximate but Redo has no closer match.

## 2026-05-08 — Migrated SMS templates explicitly emit `autoShortenLinks: false`
Operator preference is "off by default". Klaviyo's `shorten_links` is similarly opt-in, so most Klaviyo SMS templates have it false. But Redo's mongoose schema declares `autoShortenLinks: { default: true, required: true }` — omitting the field at create time → mongoose backfills `true` → migrated templates land with shortening ON. **Decision:** `parser.ts` always emits `autoShortenLinks: msg.shorten_links === true` on the SMS placeholder; `import-rpc.ts` always passes the value through (including `false`). Klaviyo's intent gets mirrored faithfully. Out of scope: changing Redo's UI default for net-new SMS templates (would be a redoapp PR; defer until operator asks).

## 2026-05-11 — Admin identity locked to one browser per slot via random claim token
Two-admin model (Austin + Michael) needed a hard lock so an invited third Replit user can't claim a slot by accident. **Decision:** new `admin_claims(user_name PK, claim_token, claimed_at)` table. First browser to pick a name via the identity modal: server generates a 32-byte random token, INSERTs the row, mirrors the token into an HttpOnly `admin_claim` cookie. Subsequent visits prove ownership by presenting the matching cookie. New `requireFullAdmin` gate requires admin_token cookie AND a valid claim — identity endpoints keep the lighter `requireAdmin` so the modal can still complete the claim flow once any claim exists. Reset is psql-only on purpose (no API). Race-safe via `INSERT … ON CONFLICT DO NOTHING` followed by a re-read; whichever INSERT wins the race owns the slot. Local dev (no `DATABASE_URL`) bypasses the claim check so the existing dev flow keeps working without ceremony.

## 2026-05-11 — Mine filter on admin Stores includes null `created_by` (legacy) rows
Stores created before admin identity tracking (migration 007) have `created_by IS NULL`. The Mine filter compared `s.createdBy === currentUser` — null didn't match either Michael or Austin, so legacy stores (e.g. Gay Pride Apparel) vanished from both views after the Mine filter started actually filtering. **Decision:** treat null-creator stores as "unclaimed, visible to both" — include them under Mine for everyone. Subtitle reads "created by X or unclaimed". Avoids the "where did my store go?" support burden. Alternative considered: add a "Claim" button on null-creator cards. Deferred — null-as-shared is fine until it isn't.

## 2026-05-11 — View-as preview is read-only via `?preview=1`, not impersonation
Michael wanted to see what Dennis sees. **Decision:** admin TopBar has separate "View as Dennis | Toby" links that open `/?as=<name>&preview=1`. Server-side nothing changes. Client-side, `?preview=1` makes the assist UI: (a) render a top banner "Previewing as <name> · read-only", (b) short-circuit `onSaveNote` / `onToggleDone` so writes don't fire. Rest of the UI behaves normally — checkboxes, search, drag-drop all work visually; only persistence is disabled. Separate "Assist ↗" link in TopBar opens `/?as=<adminUser>` (no preview) for the case where Michael wants to write as himself. Trade-off: preview is trust-based — there's no DB-side check, and a determined user could strip `&preview=1` from the URL. Acceptable since the audience is two trusted admins. Defaults the assist scope to "all" in preview so Michael sees every brand, not just brands the previewed assistant has engaged with.

## 2026-05-11 — Per-user brand-card priority via random-access table, drag-reorder in UI
Each assistant (and admin via `?as=<self>`) drags brand cards into priority order; persists per user. **Decision:** `card_priority(user_name, store_id, position)` table (migration 008). Read path: client fetches stores list AND saved order in parallel, then merges client-side — priority rows first in saved order, then any new (since-last-reorder) stores fall through in the server's default `last_imported_at DESC`. Write path: on drag-end, debounced 250ms POST replaces ALL rows for that user in one transaction (simpler than diffing). Drag-reorder is intentionally disabled in the UI when a search query is active or the Mine filter is hiding cards — reordering against a partial view is confusing and a small hint surfaces in that state. HTML5 native drag-drop (no extra deps).

## 2026-05-11 — Notes shape upgrade: structured `{text, author, savedAt}` + legacy string coexist
Per-item notes needed an author for the assist UI to attribute who wrote what. Existing admin notes were bare strings in `jobs.notes[itemId]`. **Decision:** evolve in place — the JSONB cell now holds either a legacy string OR a `{text, author?, savedAt}` object. Every reader (admin troubleshoot panel, bundle exporter, assist UI, /api/me responses) funnels through a `coerceNote` helper that normalizes both shapes to `{text, author, savedAt}`. The Toby panel keeps writing bare strings (no author on admin notes pre-identity-PR; now writes the picked Austin/Michael name as author). Cleanest path forward without backfilling — the structured shape becomes the only one over time as old strings get rewritten.

## 2026-05-11 — `/assist` reads `imported_items` (denormalized), not job_events scan
Brand picker needs a flat list of stores with N imported items each, deduped across re-imports. **Decision:** new `imported_items(store_id, store_name, job_id, item_id, item_type, name, state, imported_at, email_count)` table (migration 004 + 005 + 006 column adds). Write path: hook on the existing `exported` / `flow_imported` / campaign-`imported` event emit points; idempotent on `(store_id, item_id, job_id)`. Read path: one indexed query with `DISTINCT ON (item_id)` for latest-per-item dedup. Alternative considered: scan `job_events` JSONB on every brand-picker load. Rejected — too expensive at the read frequency the assist surface gets. Trade-off: requires the dev's Replit DB to actually have the table (added a 5-line CLAUDE.md note about not approving Replit's auto schema-diff prompt that proposes destructive DROPs).

## 2026-05-11 — `/assist` external surface stays on Replit, not Vercel
Initially considered a separate Vercel deployment for the assist UI so external assistants without Replit accounts could access it. Built end-to-end ([#52](https://github.com/MCHammer-12/mime/pull/52) closed unmerged). **Decision:** abandoned in favor of inviting assistants directly to the Replit Private Deployment. Vercel approach would have needed: separate `vercel/` directory with serverless functions, `@neondatabase/serverless` driver, URL-token gate, dual-deploy management. None of that was needed once we realized Replit invites are cheap and the existing fence works. Vercel branch kept around in case the assistant pool grows past Replit invite limits — code is at `feature/vercel-assist-deploy`.

## 2026-05-11 — Klaviyo key + Redo JWT are plain-text inputs (no password mask)
Operator pastes these repeatedly while working through merchants. Masking added friction without security benefit — the admin route is already gated by an obscure URL + cookie + claim, and the screens aren't shoulder-surfed. **Decision:** drop the password input + show/hide toggle entirely on both the Add/Edit-store modal and the legacy CredentialsBar. JWT field gains an "expired — paste fresh" red indicator and a "couldn't read store ID" hint when the token isn't a JWT (the previous silent-disabled-Save bug).

## 2026-05-07 — Saved Templates uses a different RPC, not a flag
Imported templates were all landing in Redo's "Previous emails" tab, not "Saved templates", because mime called `createEmailTemplate` for every import. Investigation: `EmailTemplate` and `SavedEmailTemplate` are two separate MongoDB collections in redoapp (`redo/model/src/saved-email-template.ts:6,12`). Saved templates is `{ _id, template: EmailTemplate, templateName, team, source, ... }` — a thin wrapper around the same EmailTemplate payload, written into a different collection by `createSavedEmailTemplate` RPC. The "Saved templates" UI tab filters to `source: "saved"` (other values `forwarded`/`uploaded` are for separate UX). **Decision:** add `asSavedTemplate?: boolean` opt to `importTemplateRpc`. Standalone template imports + campaign imports pass `true` → wrapper doc. **Flow-attached template placeholders KEEP the original `createEmailTemplate` path** because flow `send_email.templateId` references EmailTemplate `_id` at send time, not SavedEmailTemplate. Splitting per call site instead of inside `importTemplateRpc` means the caller decides intent — the function stays single-purpose.

## 2026-05-06 — Drop, don't WAIT-stub, un-translatable flow actions
Klaviyo flow actions with no Redo equivalent (`update-profile`, `list-update`, `target-date`, heavily-unmapped `send-webhook`) were emitted as 0-min WAIT stubs with `customTitle: "TODO: X (rebuild manually)"`. Live merchants reported having to clean up these stubs by hand after every import, and called them "manual activation blocks they want to avoid adding." **Decision:** drop the action entirely from the imported flow and re-stitch the predecessor's `nextId` past the dropped id. Done via a `dropAction(redirectTo)` sentinel from `convertAction`; a transitive resolver in `parseFlow` rewrites every `nextId` / `nextTrueId` / `nextFalseId` / `variants[].nextId` referencing the dropped id. `send-sms` is exempt — body content is real merchant intent, falls back to WAIT stub only when SMS body is empty (AI-content templates). `ab-test` is also special: extract `data.main_action` (Klaviyo's "winning" send-email is embedded there) and emit it as a normal email step. Supersedes prior `feedback_skipped_action_mappings` policy. Captured in memory `feedback_drop_unsupported_actions`.

## 2026-05-06 — System fonts substitute to Redo system fonts at parse time
Pretty Cult template uses `New York` (Apple's system serif, not on Google Fonts) and `Baskerville` (Linotype proprietary). Auto-upload via `buildFontPlan` returned `available: false` for both. Block-level fontFamily was set to literal "New York" / "Baskerville", which Redo's brand kit doesn't know about, so Quill falls back to browser default — visually wrong. Three options weighed: (A) substitute to Redo system serif, (B) substitute to closest Google Fonts equivalent (Libre Baskerville / Spectral) and auto-upload, (C) accept user-uploaded font files (not in scope). **Decision: (A).** User asked for "just substitute" — simpler, no upload, renders consistently across all email clients (Gmail/Outlook/webmail). Mapped both to Georgia (screen-optimized closest of Redo's allowed system serifs). Implementation: `SYSTEM_FONT_SUBSTITUTIONS` map in `fonts.ts`, applied at two layers — block-level via `parseFontFamily`, inline span declarations via new `substituteSystemFontsInHtml` HTML pre-pass in `text.ts`. Drops the system-font name from fallback chains entirely (dedupes against existing entries) so the substitute is consistently used.

## 2026-05-06 — Static product blocks emit Products block with `_pendingProducts`, not image+title columns

## 2026-05-06 — WCAG contrast guard for text + link colors
Initial dark-mode text color heuristic only fired when ALL color sources (linkStyle, inheritedLink, divStyle) were unset → swap to white if section bg dark. ## 2026-04-30 — Trigger recovery: re-parse with `forcedTrigger` override, not mid-import patching

## 2026-04-21 — Package E2: single-path inline-coupon transform, no LLMClient abstraction
Klaviyo inline `{% coupon_code 'X' %}` in body copy has no Redo equivalent. ## 2026-04-14 — Products block: dynamic-only MVP, filter defaults inferred from template HTML
Klaviyo `kl-product` blocks come in two flavors: static (hardcoded image URL + title + CTA, typically pointing at Amazon or stored Shopify URLs) and dynamic (liquid `{% if feeds.|index:N %}` + `{{ item.* }}` driven by a Klaviyo server-side product feed). Redo's `interactive-cart` block expects `productSelectionType: "dynamic"` with a `recommendedProductFilterId` pointing to a separately-created `ProductFilter` document, or `"manual"` with explicit `{ productId, variantId }` pairs in `manuallySelectedProducts`. Decision: MVP handles dynamic only — parser emits full `interactive-cart` with a `_pendingFilter: ProductFilterDoc` that the executor POSTs to `createProductFilter` and swaps for an ID before writing the template. Static blocks keep the existing COLUMN-of-images decomposition with a warning, because resolving static URLs → Shopify product IDs is a separate, deferred concern.

Filter defaults are chosen from template HTML alone because **Klaviyo's Templates API does not expose block-level feed config** — only rendered HTML with liquid tags. Heuristic: if the template anywhere contains `{% for <var> in (event.extra.line_items|items) %}`, use the Cart Item filter (`products_added_to_cart`, sort price desc, last 90 days, inventory > 0) and set `schemaFieldName: "cartContext"` on the block. Otherwise use the Best Sellers filter (`best_sellers`, sort random via omitted `sortBy`, inventory > 0). Collection-based filters cannot be inferred from HTML and will silently get the default — the importer must surface per-template warnings so Michael can retarget manually after import. `imageObjectFit: "cover"` and `imageAspectRatio: undefined` (Auto) chosen as defaults per Michael.
