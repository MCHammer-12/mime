# Decisions

## 2026-05-11 — Admin identity locked to one browser per slot via random claim token
Two-admin model (Austin + Michael) needed a hard lock so an invited third Replit user can't claim a slot by accident. **Decision:** new `admin_claims(user_name PK, claim_token, claimed_at)` table. First browser to pick a name via the identity modal: server generates a 32-byte random token, INSERTs the row, mirrors the token into an HttpOnly `admin_claim` cookie. Subsequent visits prove ownership by presenting the matching cookie. New `requireFullAdmin` gate requires admin_token cookie AND a valid claim — identity endpoints keep the lighter `requireAdmin` so the modal can still complete the claim flow once any claim exists. Reset is psql-only on purpose (no API). Race-safe via `INSERT … ON CONFLICT DO NOTHING` followed by a re-read; whichever INSERT wins the race owns the slot. Local dev (no `DATABASE_URL`) bypasses the claim check so the existing dev flow keeps working without ceremony.

## 2026-05-11 — Mine filter on admin Stores includes null `created_by` (legacy) rows
Stores created before admin identity tracking (migration 007) have `created_by IS NULL`. The Mine filter compared `s.createdBy === currentUser` — null didn't match either Michael or Austin, so legacy stores (e.g. Gay Pride Apparel) vanished from both views after the Mine filter started actually filtering. **Decision:** treat null-creator stores as "unclaimed, visible to both" — include them under Mine for everyone. Subtitle reads "created by X or unclaimed". Avoids the "where did my store go?" support burden. Alternative considered: add a "Claim" button on null-creator cards. Deferred — null-as-shared is fine until it isn't.

## 2026-05-11 — View-as preview is read-only via `?preview=1`, not impersonation
Michael wanted to see what Dennis sees. **Decision:** admin TopBar has separate "View as Dennis | Toby" links that open `/?as=<name>&preview=1`. Server-side nothing changes. Client-side, `?preview=1` makes the assist UI: (a) render a top banner "Previewing as <name> · read-only", (b) short-circuit `onSaveNote` / `onToggleDone` so writes don't fire. Rest of the UI behaves normally — checkboxes, search, drag-drop all work visually; only persistence is disabled. Separate "Assist ↗" link in TopBar opens `/?as=<adminUser>` (no preview) for the case where Michael wants to write as himself. Trade-off: preview is trust-based — there's no DB-side check, and a determined user could strip `&preview=1` from the URL. Acceptable since the audience is two trusted admins. Defaults the assist scope to "all" in preview so Michael sees every brand, not just brands the previewed assistant has engaged with.

## 2026-05-11 — Per-user brand-card priority via random-access table, drag-reorder in UI
Each assistant (and admin via `?as=<self>`) drags brand cards into priority order; persists per user. **Decision:** `card_priority(user_name, store_id, position)` table (migration 008). Read path: client fetches stores list AND saved order in parallel, then merges client-side — priority rows first in saved order, then any new (since-last-reorder) stores fall through in the server's default `last_imported_at DESC`. Write path: on drag-end, debounced 250ms POST replaces ALL rows for that user in one transaction (simpler than diffing). Drag-reorder is intentionally disabled in the UI when a search query is active or the Mine filter is hiding cards — reordering against a partial view is confusing and a small hint surfaces in that state. HTML5 native drag-drop (no extra deps).

## 2026-05-11 — Notes shape upgrade: structured `{text, author, savedAt}` + legacy string coexist
Per-item notes needed an author for the assist UI to attribute who wrote what. Existing admin notes were bare strings in `jobs.notes[itemId]`. **Decision:** evolve in place — the JSONB cell now holds either a legacy string OR a `{text, author?, savedAt}` object. Every reader (admin troubleshoot panel, bundle exporter, assist UI, /api/me responses) funnels through a `coerceNote` helper that normalizes both shapes to `{text, author, savedAt}`. The Toby panel keeps writing bare strings (no author on admin notes pre-identity-PR; now writes the picked Austin/Michael name as author). Cleanest path forward without backfilling — the structured shape becomes the only one over time as old strings get rewritten.

## 2026-05-11 — `/assist` reads `imported_items` (denormalized), not job_events scan
Brand picker needs a flat list of stores with N imported items each, deduped across re-imports. **Decision:** new `imported_items(store_id, store_name, job_id, item_id, item_type, name, state, imported_at, email_count)` table (migration 004 + 005 + 006 column adds). Write path: hook on the existing `exported` / `flow_imported` / campaign-`imported` event emit points; idempotent on `(store_id, item_id, job_id)`. Read path: one indexed query with `DISTINCT ON (item_id)` for latest-per-item dedup. Alternative considered: scan `job_events` JSONB on every brand-picker load. Rejected — too expensive at the read frequency the assist surface gets. Trade-off: requires the dev's Replit DB to actually have the table (added a 5-line CLAUDE.md note about not approving Replit's auto schema-diff prompt that proposes destructive DROPs).

## 2026-05-11 — Klaviyo key + Redo JWT are plain-text inputs (no password mask)
Operator pastes these repeatedly while working through merchants. Masking added friction without security benefit — the admin route is already gated by an obscure URL + cookie + claim, and the screens aren't shoulder-surfed. **Decision:** drop the password input + show/hide toggle entirely on both the Add/Edit-store modal and the legacy CredentialsBar. JWT field gains an "expired — paste fresh" red indicator and a "couldn't read store ID" hint when the token isn't a JWT (the previous silent-disabled-Save bug).

## 2026-05-07 — Saved Templates uses a different RPC, not a flag
Imported templates were all landing in Redo's "Previous emails" tab, not "Saved templates", because mime called `createEmailTemplate` for every import. Investigation: `EmailTemplate` and `SavedEmailTemplate` are two separate MongoDB collections in redoapp (`redo/model/src/saved-email-template.ts:6,12`). Saved templates is `{ _id, template: EmailTemplate, templateName, team, source, ... }` — a thin wrapper around the same EmailTemplate payload, written into a different collection by `createSavedEmailTemplate` RPC. The "Saved templates" UI tab filters to `source: "saved"` (other values `forwarded`/`uploaded` are for separate UX). **Decision:** add `asSavedTemplate?: boolean` opt to `importTemplateRpc`. Standalone template imports + campaign imports pass `true` → wrapper doc. **Flow-attached template placeholders KEEP the original `createEmailTemplate` path** because flow `send_email.templateId` references EmailTemplate `_id` at send time, not SavedEmailTemplate. Splitting per call site instead of inside `importTemplateRpc` means the caller decides intent — the function stays single-purpose.

## 2026-05-06 — Drop, don't WAIT-stub, un-translatable flow actions
Klaviyo flow actions with no Redo equivalent (`update-profile`, `list-update`, `target-date`, heavily-unmapped `send-webhook`) were emitted as 0-min WAIT stubs with `customTitle: "TODO: X (rebuild manually)"`. Live merchants reported having to clean up these stubs by hand after every import, and called them "manual activation blocks they want to avoid adding." **Decision:** drop the action entirely from the imported flow and re-stitch the predecessor's `nextId` past the dropped id. Done via a `dropAction(redirectTo)` sentinel from `convertAction`; a transitive resolver in `parseFlow` rewrites every `nextId` / `nextTrueId` / `nextFalseId` / `variants[].nextId` referencing the dropped id. `send-sms` is exempt — body content is real merchant intent, falls back to WAIT stub only when SMS body is empty (AI-content templates). `ab-test` is also special: extract `data.main_action` (Klaviyo's "winning" send-email is embedded there) and emit it as a normal email step. Supersedes prior `feedback_skipped_action_mappings` policy. Captured in memory `feedback_drop_unsupported_actions`.

## 2026-05-06 — System fonts substitute to Redo system fonts at parse time
Pretty Cult template uses `New York` (Apple's system serif, not on Google Fonts) and `Baskerville` (Linotype proprietary). Auto-upload via `buildFontPlan` returned `available: false` for both. Block-level fontFamily was set to literal "New York" / "Baskerville", which Redo's brand kit doesn't know about, so Quill falls back to browser default — visually wrong. Three options weighed: (A) substitute to Redo system serif, (B) substitute to closest Google Fonts equivalent (Libre Baskerville / Spectral) and auto-upload, (C) accept user-uploaded font files (not in scope). **Decision: (A).** User asked for "just substitute" — simpler, no upload, renders consistently across all email clients (Gmail/Outlook/webmail). Mapped both to Georgia (screen-optimized closest of Redo's allowed system serifs). Implementation: `SYSTEM_FONT_SUBSTITUTIONS` map in `fonts.ts`, applied at two layers — block-level via `parseFontFamily`, inline span declarations via new `substituteSystemFontsInHtml` HTML pre-pass in `text.ts`. Drops the system-font name from fallback chains entirely (dedupes against existing entries) so the substitute is consistently used.

## 2026-05-06 — Static product blocks emit Products block with `_pendingProducts`, not image+title columns
Klaviyo "static product block" (merchant-picked specific products in their grid editor) was rendered as image+title ColumnBlocks with a warning telling the merchant to manually replace via Shopify picker. Research into Redo's Products block (`redo/model/src/email-template.ts:412-451`): there's no "specific products" *filter*. Pinning is at the BLOCK level via `productSelectionType: "static"` + `manuallySelectedProducts: [{productId, variantId}]`. Auto-resolution from product names → IDs requires Shopify search, which is in redoapp (`redo/shopify/storefront/src/products.ts:795`) but not exposed to mime via RPC. **Decision:** mime emits the Products block with `productSelectionType: "static"` + new non-prod field `_pendingProducts: [{name}]` (mirrors existing `_pendingFilter` pattern). Importer-side resolution via Shopify Storefront/Admin search is a follow-up in `redo/manage/src/import-klaviyo-templates.ts`. Static blocks in AC context (cart-loop signal in doc) are upgraded to dynamic Products + `CART_ITEM_FILTER` instead — pinned products discarded with a warning, runtime cart items take over at send.

## 2026-05-06 — WCAG contrast guard for text + link colors
Initial dark-mode text color heuristic only fired when ALL color sources (linkStyle, inheritedLink, divStyle) were unset → swap to white if section bg dark. Missed common cases: divStyle.color set to a value that still has poor contrast (Klaviyo's CSS-default `#15c` blue link rendered through our parser as `parseColor("#15c")` → `#15c` even on a black bg). User reported "link color is blue, should be white" — that's exactly this case. **Decision:** replace with WCAG-style contrast check. New `contrastRatio(fg, bg)` helper. If resolved color contrasts below 3:1 against section bg (universally unreadable, well under WCAG AA Large), swap to whichever of black/white reads better. Applied to text + link in `text.ts` and to product-block titles in `product.ts`. Inline span colors override per-segment so this only adjusts the block-level default.

## 2026-04-30 — Klaviyo footer tags: map when Redo has equivalent, drop+warn otherwise
Live merchants surface a mix of Klaviyo Liquid in footer text: `{% unsubscribe %}` (and arg + link variants), `{% web_view %}`, `{% manage_preferences %}`, `{{ first_name }}`, `{{ shop.name }}`, `{{ email_preference_url }}`. After cross-referencing `redoapp/redo/model/src/email-builder/shared-schema-fields.ts` and `redoapp/redo/model/src/advanced-flow/schemas/marketing/marketing.ts`, three of these have direct Redo equivalents (`unsubscribe_link`, `view_in_browser_link`, `customer_first_name` and the rest of the `customer_*` schema fields). The others (`manage_preferences`, `email_preference_url`) have no Redo schema — preference-center links don't exist in the Redo email runtime. Decision: in `transform.ts` `substituteTextVars`, MAP whenever Redo has an equivalent (rewrite the tag and log a substitution), DROP+WARN otherwise (strip the wrapping `<a>` plus one adjacent " | " / " or " separator and emit a warning so the merchant sees the data loss). Drop semantics are atomic with the separator strip so a downstream cleanup pass doesn't have to guess (an earlier "remove all stranded separators" cleanup ate valid pipe-separators between unrelated anchors — fixed by limiting the pass to consecutive-separator collapse only).

## 2026-04-30 — Troubleshoot bundle: zip download, free-text notes, multi-select per job
Live import bugs need offline debugging with the original Klaviyo source + the Redo output + parse-result + a free-text "what's wrong" annotation. Two delivery options weighed: (a) paste-ready markdown the user copies into chat, or (b) zip file the user downloads + attaches. Picked (b): markdown truncates Klaviyo HTML for non-trivial templates and loses the .redo-template.json structure; a zip preserves everything verbatim and Claude reads attached files directly. Notes are free-text per template/flow (no severity tags — added complexity without proportional value at this stage), persisted to a new `jobs.notes` JSONB column (DB migration 002). Multi-select via checkboxes + a single "Export zip (N)" button per job card. Implemented in `src/migrate/bundle.ts` using `archiver` (only new dep), streams directly to res so large bundles don't buffer in memory.

## 2026-04-30 — Trigger recovery: re-parse with `forcedTrigger` override, not mid-import patching
Klaviyo flows with custom-event metric triggers (or other unmapped trigger types) used to fail with "skipped: unresolvable trigger". Two options to fix: (a) patch the `AdvancedFlow` JSON in-place after a partial parse, swapping in the user-chosen trigger, or (b) re-parse the original Klaviyo flow with a `forcedTrigger` override that bypasses the auto-resolver. Picked (b): re-running the full parse with the override goes through the same code path that handles abandonment skip-conditions, treeify, and step assembly — there's no risk of partial state from a half-parsed flow. The cost is a second parse pass (cheap; flow JSON is small). Implementation: `parseFlow` accepts `opts.forcedTrigger?: TriggerResolution`; when omitted and auto-resolution fails, `ParseResult.skipped` carries `recoverable: true` + the original `klaviyoTrigger` so the caller can render context for the picker prompt. Picker options live in `src/flow/marketing-trigger-options.ts` mirroring `redoapp/redo/model/src/advanced-flow/triggers.ts schemaTypeConfigs` (13 marketing triggers). Server flow-import detects recoverable skip → emits `needs_input` (type=choice) → re-parses → imports normally.

## 2026-04-30 — Klaviyo filter strings: standardize on double quotes
Klaviyo's JSON:API filter spec accepts both single and double quotes per-RFC, but in practice some merchant accounts reject the single-quoted form with HTTP 400. The codebase had two patterns: flow-messages used `equals(flow-action.id,"<id>")` (working) and campaigns used `equals(messages.channel,'email')` (failing on at least one merchant). Decision: standardize on double quotes everywhere. Same fix applied to `src/extract-campaigns.ts`. Same change in `src/klaviyo.ts` to lift Klaviyo's `errors[0].detail/title/code` to the front of the thrown error message so a truncated UI display surfaces the actionable reason.

## 2026-04-21 — Package E2: single-path inline-coupon transform, no LLMClient abstraction
Klaviyo inline `{% coupon_code 'X' %}` in body copy has no Redo equivalent. The transform now unconditionally rewrites the sentence and inserts a placeholder `DiscountBlock` directly below — single deterministic structure, no branching on "maybe keep the variable". The AI rewrite is written against that invariant ("a discount block will be inserted below") so prompt output is predictable. When AI is off (`SKIP_AI=1` or no API key), a rule-based regex strips the common `"USE CODE X FOR N% OFF"` phrase; if the phrase doesn't match, the Jinja tag stays in the text and a discount block is still appended, so every inline-coupon template produces a discount block either way. Column cells are the one exception: `ColumnBlock.columns` is one slot per column, so inserting a sibling discount block isn't possible — we rewrite the text but log a warning and skip the insertion. The Replit deployment detail (auto-provisioned `AI_INTEGRATIONS_*` env vars) means the stock Anthropic SDK works on both Replit and locally with a single `new Anthropic({ baseURL, apiKey })` instantiation — no `LLMClient` wrapper needed; the SDK *is* the abstraction. Discount-object creation + `discountId` wiring stays in the redoapp import executor (team-scoped API auth, discount already exists in the merchant's Redo account).

## 2026-04-14 — Static product blocks: hybrid with Shopify GraphQL resolver (revised)
The earlier "dynamic-only MVP" decision left static product blocks decomposed as a ColumnBlock of ImageBlocks. New data: static blocks are ~50% of Quikcamo's product-containing templates (170 of 331), and ~74% (126) of those link to Shopify URLs with extractable product handles. Revised: the executor resolves static product handles to Shopify productId/variantId via the merchant's existing Shopify GraphQL connection (ShopifyProvider.searchProducts), builds a real PRODUCTS block with `productSelectionType: "manual"` + `manuallySelectedProducts`, and falls back to Column of Images only when the lookup fails (product deleted, non-Shopify URL, or handle mismatch). No separate cache — live Shopify query per handle, batched for rate limits. Post-import report flags fallbacks for merchant review. Research confirmed Shopify access is available from `redo/manage/` scripts via `ShopifyProvider` with the team's stored access token.

Originally planned to use AI for: inline coupon sentence rewrites, discount amount inference, image-as-button detection, font ambiguity resolution. Revised to AI-minimal: the only remaining AI use is inline coupon sentence rewriting (where LLM genuinely outperforms regex/prompts). All other touchpoints use deterministic rules + user prompts at the config/preflight phase. ## 2026-04-14 — Font provisioning: preflight block, no programmatic upload
Research confirmed we CAN programmatically upload fonts to the merchant's Redo brand kit (`uploadFile` → `processFontFiles` RPC → `TeamRepo.updateBrandKit`). Decision: don't. Instead, the import script preflights the merchant's existing brand kit against the fonts required by the parsed templates; if any are missing, it blocks migration and prints an actionable list (font name, template count, Google Fonts URL where available, link to merchant's brand kit page). Rationale: no surprise uploads to the merchant's brand kit, no silent fallbacks that degrade visual fidelity, merchant retains full control. Idempotent: merchant adds fonts, re-runs script, preflight passes, migration proceeds. Documentation of the programmatic path kept in `reference_brand_kit_font_upload` memory for future use (e.g. a self-serve merchant-facing migration tool).

## 2026-04-14 — Store ID is the user-facing term for team ID
Redo's data model uses `team: ObjectId` internally (see `EmailTemplate.team`, `TeamRepo`). The merchant dashboard URL bar shows `/stores/<id>/...`. Decision: CLI flags + user-facing prompts use "store ID"; internal code passes it as `team`. They're the same MongoDB ObjectId. Avoids merchant confusion ("what's my team ID?" has no answer; "what's my store ID?" is the URL).

## 2026-04-14 — Products block: dynamic-only MVP, filter defaults inferred from template HTML
Klaviyo `kl-product` blocks come in two flavors: static (hardcoded image URL + title + CTA, typically pointing at Amazon or stored Shopify URLs) and dynamic (liquid `{% if feeds.|index:N %}` + `{{ item.* }}` driven by a Klaviyo server-side product feed). Redo's `interactive-cart` block expects `productSelectionType: "dynamic"` with a `recommendedProductFilterId` pointing to a separately-created `ProductFilter` document, or `"manual"` with explicit `{ productId, variantId }` pairs in `manuallySelectedProducts`. Decision: MVP handles dynamic only — parser emits full `interactive-cart` with a `_pendingFilter: ProductFilterDoc` that the executor POSTs to `createProductFilter` and swaps for an ID before writing the template. Static blocks keep the existing COLUMN-of-images decomposition with a warning, because resolving static URLs → Shopify product IDs is a separate, deferred concern.

Filter defaults are chosen from template HTML alone because **Klaviyo's Templates API does not expose block-level feed config** — only rendered HTML with liquid tags. Heuristic: if the template anywhere contains `{% for <var> in (event.extra.line_items|items) %}`, use the Cart Item filter (`products_added_to_cart`, sort price desc, last 90 days, inventory > 0) and set `schemaFieldName: "cartContext"` on the block. Otherwise use the Best Sellers filter (`best_sellers`, sort random via omitted `sortBy`, inventory > 0). Collection-based filters cannot be inferred from HTML and will silently get the default — the importer must surface per-template warnings so Michael can retarget manually after import. `imageObjectFit: "cover"` and `imageAspectRatio: undefined` (Auto) chosen as defaults per Michael.

## 2026-04-14 — Footer: keep as Text block, don't use Redo's FooterBlock
Klaviyo footers are text blocks containing `{% unsubscribe %}` and `{{ organization.* }}`. Original plan (per `project_klaviyo_footer_variables.md` and `TODO-SHARED-text.md` PRIORITY 0) was to convert them to Redo's `FooterBlock` with `schemaFieldName: "unsubscribeLink"` and `useTemplateAddress: true`. Prototyped end-to-end, but reversed after viewing rendered output: FooterBlock forces its own fixed copy/order ("business name / address / city-state-zip / country / Unsubscribe") with no configurable preamble, destroying Klaviyo's phrasing ("No longer want to receive these emails?") and merchant customization. Decision: keep as TextBlock and substitute the variables inline in the export pipeline — `{% unsubscribe %}` → `<a href="{{ unsubscribe_link }}">unsubscribe</a>` (Redo recognizes `{{ unsubscribe_link }}` inside text per `hasUnsubscribeLink` in web email-builder utils), and org variables → merchant-provided values from Klaviyo Accounts API (or user prompt fallback). Placeholder text is unacceptable — if API call fails, migration must prompt before proceeding. No new block type; no FooterBlock added to `types.ts`. Abandoned files: `src/parser/blocks/footer.ts`, `src/renderer/blocks/footer.tsx`.

## 2026-04-14 — Discount: split text block, not a hybrid
Klaviyo `{% coupon_code 'Name' %}` can appear either standalone (its own line between `<br/>`s) or inline in a sentence. Redo has no inline coupon primitive — a coupon must be its own DiscountBlock with an associated Redo discount object. Decision: deterministic parser handles standalone coupons only, splitting the surrounding text block into `[text, discount, text]`; inline coupons are left in the text block for the migration's downstream AI-rewrite pass to handle. Michael explicitly confirmed this over a "text/discount hybrid" block, which was considered and rejected because the prod schema offers no such hybrid.

## 2026-04-14 — Socials icon color: lossy mapping to prod enum
Prod Redo's `SocialIconColor` enum allows only `black`/`white`/`gray`. Local `types.ts` still lists `original` but it fails prod Zod validation. Klaviyo's `/default/` CDN path serves colorful branded icons with no exact Redo equivalent. Decision: `/default/` → `black`. Michael confirmed exact icon color match is not required for migrations as long as background, URLs, and padding are correct.

## 2026-04-14 — Klaviyo hlb-wrapper → Redo ImageBlock (not HeaderBlock)
Klaviyo's Header/Logo Bar (`hlb-wrapper`, used by 91% of templates) combines a logo image with optional menu links. Redo has a native `HeaderBlock`, but it auto-pulls the logo from the team's brand kit — unreliable for migrations where the brand kit may not be set. Decision: `parseHeaderBlock` emits an `ImageBlock` for the logo portion (width preserved via calculated horizontal inner padding so a 300px Klaviyo logo renders at ~300px in the 600px Redo email), and `parseMenuFromHeader` separately emits a `MenuBlock` for nav links. The `header.tsx` renderer is now dead code for Klaviyo migrations.
